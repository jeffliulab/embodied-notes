# 感知与定位

> *Perception & Localization — 自动驾驶的"眼睛"与"GPS 加强版"：把多路传感器原始信号变成一个统一、可被规划消费的 BEV/占据世界模型，并知道自车在地图里的厘米级位置.*
>
> **难度**：Intermediate
> **前置知识**：[自动驾驶范式与方法谱系](../01_Foundations/自动驾驶范式与方法谱系.md)、[SLAM：同时定位与建图](../../00_Robotics/02_Classical_Robotics/SLAM.md)、[坐标系](../../00_Robotics/02_Classical_Robotics/坐标系.md)
> **后续阅读**：[端到端驾驶](../05_Learning_E2E/端到端驾驶.md)、[数据集与闭环评测](../06_Data_Eval/数据集与闭环评测.md)

---

感知（perception）负责回答"**周围有什么、在哪里、往哪走**"，定位（localization）负责回答"**我在世界/地图的哪个位置、什么朝向**"。在[范式谱系](../01_Foundations/自动驾驶范式与方法谱系.md)的模块化流水线里，二者是最前端的两块；即便在 UniAD 这类端到端可微系统中，它们也以**显式 BEV/占据中间表示**的形式被保留。本篇沿"从单传感器到统一 BEV 世界模型"这条主线，把相机 3D 检测、LiDAR 检测、占据栅格、多传感器融合、在线建图与组合导航/SLAM 配准串成一条逻辑链。

## 1. 感知任务总览（2D/3D 检测、语义/全景分割、跟踪）

自动驾驶感知不是一个任务，而是一组互补任务，按"输出表示"分类最清楚：

| 任务 | 输出 | 典型用途 | 度量 |
|---|---|---|---|
| **2D 检测** | 图像平面 bbox + 类别 | 红绿灯、交通标志识别 | mAP@IoU |
| **3D 检测** | 自车坐标系下的 9-DoF 框 $(x,y,z,w,l,h,\theta,v_x,v_y)$ | 障碍物、给预测/规划喂框 | nuScenes mAP / NDS |
| **语义分割** | 每像素/每体素类别 | 可行驶区域、车道、地形 | mIoU |
| **全景分割** | 语义 + 实例（stuff + things） | 同时要"是什么"和"哪一个" | PQ |
| **多目标跟踪 (MOT)** | 跨帧一致的 track ID + 轨迹 | 给预测提供历史、速度估计 | AMOTA / MOTA / IDF1 |
| **占据预测** | 体素级 (占据 + 语义[+流]) | 长尾几何、未知障碍物 | mIoU / RayIoU |

关键观念：现代量产栈的输出**统一到 BEV（bird's-eye-view，鸟瞰俯视）平面或 3D 体素**上，而不是停留在透视图像里。原因有三——规划在 BEV 上做最自然；多相机/多传感器只有投到同一坐标系才能融合；时间维（速度、历史轨迹）在 BEV 上是平移而非透视形变。3D 检测里的 $v_x,v_y$ 体现了 nuScenes 等基准要求**联合估计速度**，因为下游预测/规划直接消费它。

跟踪经典范式是 **tracking-by-detection**：先逐帧检测，再用运动模型（卡尔曼滤波）+ 数据关联（匈牙利匹配 / 贪心）把检测连成 track。CenterPoint 利用其 center heatmap 预测的速度向量做帧间贪心关联，几乎免费拿到 MOT。

## 2. 相机 3D 检测与 BEV 表示

相机便宜、语义丰富，但单目缺深度——3D 检测的核心难点是**从 2D 透视图恢复 3D 度量信息**。

### 2.1 透视 → BEV 的几何

针孔相机模型把 3D 点 $X_c=(X,Y,Z)$ 投到像素：

$$
\lambda \begin{bmatrix} u \\ v \\ 1 \end{bmatrix} = K\,[R\mid t]\, \begin{bmatrix} X \\ Y \\ Z \\ 1 \end{bmatrix},\qquad K=\begin{bmatrix} f_x & 0 & c_x \\ 0 & f_y & c_y \\ 0 & 0 & 1 \end{bmatrix}
$$

深度 $\lambda=Z$ 在投影时被**除掉**——这就是为什么单张图反投影回 3D 是病态的：一个像素对应一整条射线 $\{X_c=\lambda K^{-1}[u,v,1]^\top\}$。把图像特征送到 BEV，本质就是要**沿这条射线放置/聚合特征**。两条主流路线：

- **正向（lift / push）**：给每个像素估一个**深度分布**，沿射线把特征"撒"到 BEV 网格——LSS。
- **反向（pull / sample）**：在 BEV 网格上预定义 query，反投到各图像采样特征——BEVFormer。

### 2.2 LSS：Lift-Splat-Shoot

LSS（Philion & Fidler, ECCV 2020）三步：

1. **Lift**：对每个像素，CNN 输出上下文特征 $c\in\mathbb{R}^{C}$ 和一个离散**深度概率分布** $\alpha\in\Delta^{|D|}$（在预设深度 bin $D=\{d_1,\dots,d_k\}$ 上）。该像素在深度 $d_j$ 处的特征为外积 $\alpha_j\, c$，于是单像素被"抬"成一根 3D 视锥点云列（frustum）。这种**软深度**（不硬选一个深度）是 LSS 区别于早期方法的关键。
2. **Splat**：把所有视锥点用相机外参变到自车坐标，按 BEV 网格做 **sum-pooling**（pillar 累加）。借助累积和（cumulative-sum trick）在排序后的点上高效求和，避免显式体素循环。
3. **Shoot**：在生成的 BEV 特征图上做下游任务（分割 / 检测 / 规划"射"出轨迹）。

其优雅在于**端到端可微**——深度分布是隐式学出来的，无需深度真值监督（但有 LiDAR 深度监督会显著更准，如 BEVDepth）。

### 2.3 BEVFormer：时空注意力

BEVFormer（Li et al., ECCV 2022）走反向路线，维护一组可学习的 **BEV query** $Q\in\mathbb{R}^{H\times W\times C}$，每个 query 绑定一个 BEV 网格位置：

- **空间交叉注意力（spatial cross-attention）**：每个 BEV query 沿高度采若干 3D 参考点，用相机投影矩阵投到各相机图像，只在命中的视图上做**可变形注意力**（deformable attention）采样特征。这避免了 LSS 显式估深度，把"深度歧义"交给注意力隐式解决。
- **时间自注意力（temporal self-attention）**：把上一帧 BEV 特征按自车运动 ego-motion 对齐（warp）后，与当前 BEV query 做注意力。时间堆叠让模型能从多帧三角化出**深度与速度**，对遮挡和低速目标尤其有效。

## 3. LiDAR 检测（点云表示与 anchor-free）

LiDAR 直接给出**度量精确的稀疏 3D 点云** $\{(x,y,z,\text{intensity})\}$，没有深度歧义，但点云**无序、稀疏、密度随距离骤降**。核心设计是"如何把无序点编码成卷积/Transformer 能吃的规则张量"。

### 3.1 体素化：VoxelNet 与 PointPillars

- **VoxelNet**（Zhou & Tuzel, CVPR 2018）：把空间切成规则 3D 体素，每个体素内用 **VFE（voxel feature encoding）** 小型 PointNet 把可变数量的点编码成定长向量，再接 3D 卷积 + RPN 出框。问题是 3D 卷积**昂贵**，且 z 方向多数体素是空的。
- **PointPillars**（Lang et al., CVPR 2019）：关键工程简化——只在 **(x,y) 切柱（pillar）**，每根柱沿 z 不再细分。柱内点过 PointNet 得柱特征，散回成一张 **2D 伪图像（pseudo-image）**，然后用纯 **2D CNN** 检测。代价是丢了细 z 分辨率，但换来**数量级的速度提升**（>60 FPS），至今是实时量产的强基线。

直觉对比：VoxelNet 是"3D 体素 + 3D 卷积"，PointPillars 是"2D 柱 + 2D 卷积"——把昂贵的第三维拍扁。

### 3.2 CenterPoint：anchor-free

基于 anchor 的检测器在 3D 里很别扭：物体朝向任意，预设朝向各异、大小各异的 anchor 既贵又难匹配。**CenterPoint**（Yin et al., CVPR 2021）借鉴 2D 的 CenterNet，把检测改成**关键点（中心点）回归**：

1. 在 BEV 特征图上预测每类一张**中心 heatmap**，峰值即物体中心；
2. 在峰值位置回归子像素偏移、$z$ 高度、3D 尺寸 $(w,l,h)$、朝向 $\theta$（用 $(\sin\theta,\cos\theta)$ 避免角度环绕）和**速度** $(v_x,v_y)$；
3. 第二阶段（two-stage）用框面中心的点特征做 refine。

anchor-free 的好处：旋转目标不再需要旋转 anchor 匹配；速度头让它**自带跟踪**（用 $v$ 把当前中心投影回上一帧做贪心关联）。CenterPoint 是当前 LiDAR / 融合检测的事实标准 backbone。

## 4. Occupancy Network 占据栅格

3D 框的根本局限：它假设世界由**已知类别的刚性长方体**组成。但真实道路上有**异形与长尾几何**——掉落的轮胎、施工水马、横翻的卡车、塑料袋、伸出车斗的钢筋。框模型对"没见过类别 / 非长方体形状"的物体会漏检或乱框。

**占据栅格网络**（occupancy network，如 OccNet / TPVFormer / SurroundOcc，2023）改用**与类别无关的几何表示**：把自车周围空间离散成体素 $V\in\{0,1\}^{X\times Y\times Z}$，预测每个体素是否被占据（可再附语义、附 occupancy flow 速度）。它本质是把图像/2D 分割的思想搬到 3D 稠密体素——"这块空间能不能开过去"，无需先问"这是什么类别"。

为什么更通用：

- **长尾几何无忧**：不依赖类别先验，任何形状的障碍物只要"占了空间"就被表达，天然处理 open-set 障碍。
- **可表达自由空间与不可见**：体素可标注 free / occupied / unknown，规划能显式利用"已知可行驶空间"，对未观测区保守。
- **与规划接口自然**：碰撞检测本就是体素/栅格查询，占据图直接喂代价地图。

代价是**体素稠密、算量/显存大**，故 TPVFormer 用三视图平面（tri-perspective view）分解 $\mathcal{O}(XYZ)\to\mathcal{O}(XY+YZ+ZX)$ 近似稠密体素。特斯拉 2022 AI Day 把 Occupancy Network 作为量产视觉栈核心，把这条路线推向工业主流。可视作 §2 BEV 的 3D 推广：BEV 是把 z 拍扁的占据图，占据网络保留了 z。

## 5. 多传感器融合

没有单一传感器全能。融合的目标是**取长补短并提供冗余**（安全冗余对量产是硬需求）。

### 5.1 三种融合层次

| 融合层次 | 在哪融 | 优点 | 缺点 |
|---|---|---|---|
| **前融合 (early / data-level)** | 原始数据级（如把 LiDAR 点投到图像着色，或图像特征装饰点云）| 信息损失最小、强互补 | 需精确时空标定与同步，对标定误差敏感 |
| **特征融合 (mid / feature-level)** | 各模态编码成特征后，在 BEV/体素空间对齐拼接（BEVFusion 类）| 表达力与鲁棒性平衡，当前主流 | 需统一表示空间，工程复杂 |
| **后融合 (late / object-level)** | 各模态各自出检测框，再做框级关联/卡尔曼融合 | 模块解耦、易加传感器、单路失效可降级 | 早期丢信息，弱目标各路都漏则无法救回 |

趋势：以 **BEV 特征融合**为主干（BEVFusion 把相机 LSS 特征与 LiDAR 体素特征统一到同一 BEV 平面后拼接），辅以后融合做安全冗余兜底。

### 5.2 相机 + LiDAR + radar 互补

| 维度 | 相机 Camera | LiDAR | Radar (毫米波) |
|---|---|---|---|
| 深度/测距 | 弱（单目病态） | **强**（厘米级直接测量） | 中（米级，但远距尚可） |
| 语义/颜色/纹理 | **强**（读交通灯、标志、车道） | 弱（无颜色） | 无 |
| 角分辨率 | 高 | 高 | 低（难分辨相邻目标） |
| 直接测速 | 无（需多帧） | 无（需多帧） | **强**（多普勒直接测径向速度） |
| 恶劣天气（雨雾尘）| 差 | 中（雨雾散射衰减） | **强**（穿透好） |
| 夜间/眩光 | 差（依赖光照） | **强**（主动光） | 强 |
| 成本 | **低** | 高 | 低 |

互补逻辑很清楚：相机给语义、radar 给速度与全天候穿透、LiDAR 给精确几何。三者在 BEV 空间对齐后，弱项被他者覆盖；任一路失效（相机眩光、LiDAR 雨雾）系统仍可降级运行——这正是后融合冗余的价值。

## 6. 高精地图 HD map vs 无图方案

### 6.1 HD map：先验即"离线感知"

高精地图（HD map）以**车道级**精度（厘米级）离线存车道中心线、连接拓扑、停止线、人行横道、限速、红绿灯位置。它本质是**把一部分感知离线做好缓存**：有图时在线只需定位 + 动态障碍感知，静态语义查表即得，极大降低在线负担、提升稳定性。

代价是**重**：采集与高精度标注昂贵；地图会过期（施工、改道），需庞大更新闭环；强依赖厘米级定位把图对齐到当前帧。覆盖范围因此成了规模化瓶颈——这直接催生了无图路线。

### 6.2 无图 / 在线建图：MapTR 类

"无图"（mapless）并非不要地图，而是**在线实时建局部矢量地图**（不依赖预存 HD map）。**MapTR**（Liao et al., ICLR 2023）把在线建图建模为**矢量地图元素的集合预测**：用一组 DETR 式 query 直接输出车道线、边界、人行横道等的**有序点序列折线**（polyline），而非先分割再后处理矢量化。

技术要点：地图元素是**置换等变**的（一条折线正反两序、闭合多边形起点任意都等价），MapTR 用**点集层级的等价排列匹配**（hierarchical bipartite matching）建模这种对称性，使训练稳定、推理端到端出矢量结果。输出的矢量地图直接给规划当车道约束。

权衡：HD map = 重先验、稳定、覆盖受限；无图 = 轻、可规模化、但在线建图质量在复杂路口仍逊于精修离线图。量产常折中——主干道用轻量"导航级"地图（SD map）+ 在线建图补全车道级细节。

## 7. 定位

定位要在地图/世界坐标系下给出自车 6-DoF 位姿，量产要求**厘米级且高频、不间断**。没有单一手段同时满足，故用**互补滤波/因子图融合**。

### 7.1 GNSS + IMU 组合导航

- **GNSS**（GPS/北斗，配 RTK 差分可达厘米级）给**全局绝对**位置，但**低频**（~10 Hz）、城市峡谷多径与隧道遮挡会失锁、有跳变。
- **IMU**（加速度计 + 陀螺）**高频**（>100 Hz）、短时精确，但积分有**漂移**（误差随时间累积），单独不可用。

二者天然互补：GNSS 校正 IMU 的长期漂移，IMU 在 GNSS 失锁/低频间隙做**航位推算**（dead reckoning）平滑插值。融合用**误差状态卡尔曼滤波（ESKF）**或图优化，常再加轮速里程计。这与 [SLAM](../../00_Robotics/02_Classical_Robotics/SLAM.md) 中的状态估计同源（见其 EKF/因子图章节）。

### 7.2 LiDAR / 视觉 SLAM 配准

GNSS 失锁处（隧道、地库、城市峡谷）靠**地图配准**定位：

- **LiDAR 配准**：把当前点云与预建点云地图用 **ICP / NDT** 配准求位姿。几何精确、对光照鲁棒，是高精定位主力（NDT 把地图建成正态分布栅格，比逐点 ICP 更快更稳）。
- **视觉定位**：用 [ORB-SLAM](../../00_Robotics/02_Classical_Robotics/SLAM.md) 类特征 SLAM 或视觉重定位，把当前帧匹配到视觉地图。成本低，但对光照、纹理缺失敏感。

注意区分两个相关但不同的问题：**SLAM** 是无图时同时建图 + 定位（探索阶段）；自动驾驶量产**定位**多是**已知地图下的配准/重定位**（地图已存）。本篇定位深挖到此为止，状态估计的概率基础、回环检测、位姿图优化详见 [SLAM：同时定位与建图](../../00_Robotics/02_Classical_Robotics/SLAM.md)。

## Common Pitfalls

1. **把单目 3D 深度当可靠真值**：单目深度是学习先验，雨雾、域偏移、未见尺寸物体下系统性偏差大；安全件不应只靠单目深度。
2. **忽视 BEV 的标定/时间同步依赖**：所有"投到同一 BEV"的融合都假设外参标定与多传感器时间戳精确，标定漂移直接让框错位——这是量产最隐蔽的事故源。
3. **只用 3D 框，忽略长尾几何**：异形/未见类别障碍会被框模型漏检，占据栅格正是为此存在（见 §4）。
4. **mAP 高 ≠ 上路安全**：检测 mAP 是开环指标，对小目标/远距/被遮挡安全关键目标的召回、以及速度估计误差，往往比平均 mAP 更要命；用 nuScenes NDS（含速度/属性误差）或闭环评测更可靠。
5. **混淆 SLAM 与定位**：量产定位多是"已知地图下配准"，不是从零 SLAM；把二者当一回事会高估在线建图的可靠性需求。
6. **过度依赖 HD map**：地图过期（施工改道）时若无在线感知兜底会危险；图永远是先验，不能替代实时感知。

## 前沿趋势

1. **统一 BEV / 占据为通用世界模型**：感知输出从"框集合"转向"稠密 BEV/占据世界模型"，下游预测、规划、端到端（UniAD）共享同一表示，详见[范式谱系](../01_Foundations/自动驾驶范式与方法谱系.md)。
2. **无图化与轻地图**：MapTR 类在线矢量建图 + SD map 先验，降低对重 HD map 的依赖以实现规模化。
3. **占据流与 4D 占据**：在占据体素上加 occupancy flow（每体素速度），把"是什么/在哪"扩展到"往哪动"，模糊感知与预测边界。
4. **以视觉为主、radar 复兴**：纯视觉占据（特斯拉）压成本，同时 4D 成像 radar（带高度/角分辨）回归融合栈补全天候短板。
5. **感知-预测-规划联合可微**：检测/跟踪/占据不再独立优化，而是以最终规划目标统一反传（UniAD），缓解中间表示的信息瓶颈。
