# Vehicle Control

> *Vehicle Control — the last mile of the autonomy stack: turn the reference trajectory handed down by the planner (a curve with a speed profile) into steering angle and throttle/brake so the car tracks it to centimeter accuracy, smoothly and stably. The core difficulty: a car is a nonholonomic, second-order system with actuation delay and nonlinear tires.*
>
> **Difficulty**: Intermediate
> **Prerequisites**: [Robot Control Theory](../../00_Robotics/02_Classical_Robotics/控制理论.en.md), [MPC — Model Predictive Control](../../00_Robotics/02_Classical_Robotics/MPC.en.md), [Planning and Decision-Making](../03_Prediction_Planning/规划与决策.en.md)
> **Further reading**: [End-to-End Autonomous Driving](../05_Learning_E2E/端到端驾驶.en.md)

---

Vehicle control sits right after motion planning: the planner outputs a reference trajectory $\{(x_r,y_r,\theta_r,v_r,\kappa_r)\}$, and the controller pulls the vehicle's actual state back onto that line **in real time**. It splits into two largely independent yet ultimately coupled subtasks:

- **Longitudinal control** — regulate speed / gap, output acceleration → throttle/brake.
- **Lateral control** — regulate cross-track error / heading, output front-wheel angle → steering.

Below we start from the simplest kinematic model and progressively reach the MPC that solves both jointly in one constrained optimization.

---

## 1. Vehicle Dynamics Fundamentals

### 1.1 Kinematic Bicycle Model

Collapse the four-wheel car into one front and one rear equivalent wheel (hence "bicycle"). Take the rear-axle center as the reference point $(x,y)$, body heading $\theta$, front-wheel angle $\delta$, wheelbase $L$, rear-axle speed $v$. The **no-slip assumption** (wheels roll only along their own heading, valid at low speed) gives the state equations:

$$
\dot{x} = v\cos\theta,\qquad
\dot{y} = v\sin\theta,\qquad
\dot{\theta} = \frac{v}{L}\tan\delta,\qquad
\dot{v} = a
$$

Geometric intuition: the front-wheel angle $\delta$ sets the instantaneous turning radius $R = L/\tan\delta$, so the heading rate is $\dot\theta = v/R = (v/L)\tan\delta$. This is the foundation of every lateral-control curvature formula — **the controller computes a "curvature" $\kappa = \tan\delta / L$ and inverts it for $\delta$**.

If the reference point is at the **center of mass** rather than the rear axle, introduce the slip angle $\beta = \arctan\!\big(\tfrac{l_r}{l_f+l_r}\tan\delta\big)$:

$$
\dot{x} = v\cos(\theta+\beta),\quad
\dot{y} = v\sin(\theta+\beta),\quad
\dot{\theta} = \frac{v\cos\beta}{l_f+l_r}\tan\delta
$$

where $l_f, l_r$ are CoM-to-front/rear-axle distances. The kinematic model is accurate **only at low speed (< ~5 m/s) and small steering**, since it assumes the tires never slip.

### 1.2 Dynamic Model and Tire Slip

In high-speed cornering tires develop a **slip angle** $\alpha$ — the angle between the tire's actual velocity direction and its heading. Under a linear tire model the lateral force is proportional to slip angle:

$$
F_{y,f} = C_{\alpha f}\,\alpha_f,\qquad F_{y,r} = C_{\alpha r}\,\alpha_r
$$

$C_\alpha$ is the **cornering stiffness**. The **linear single-track dynamic model** with lateral velocity $v_y$ and yaw rate $r=\dot\theta$ as states:

$$
m(\dot{v}_y + v_x r) = F_{y,f} + F_{y,r},\qquad
I_z \dot{r} = l_f F_{y,f} - l_r F_{y,r}
$$

with slip angles approximated as $\alpha_f = \delta - \tfrac{v_y + l_f r}{v_x}$, $\alpha_r = -\tfrac{v_y - l_r r}{v_x}$. Key points:

- **Finite cornering stiffness** → at high speed the actual turning radius exceeds the kinematic prediction ("understeer"); the kinematic model biases the controller.
- **Tire-force saturation**: real tires are nonlinear (Pacejka "magic formula"); near the friction limit $F_y$ stops growing with $\alpha$ → loss of stability.
- In practice: kinematic model for low-speed parking, dynamic model for high speed / racing; MPC often switches by speed or uses the dynamic model throughout.

## 2. Longitudinal Control

Goal: track a reference speed $v_r$ (cruise) or reference gap $d_r$ (car-following / ACC). The control variable is desired acceleration $a$, mapped through an inverse longitudinal model (engine/motor map + brake) into throttle/brake.

### 2.1 PID Speed Control

The simplest and most widely used:

$$
a = K_p(v_r - v) + K_i\!\int(v_r - v)\,dt + K_d\frac{d}{dt}(v_r - v)
$$

The integral term removes steady-state error from grade / drag; the derivative term suppresses overshoot (but is sensitive to speed noise, so usually low-pass filtered). **Feedforward** is crucial: when the reference acceleration $a_r$ is known, set $a = a_r + \text{PID}$, leaving PID to handle only the residual — markedly more accurate tracking.

ACC car-following uses two loops: the outer loop computes desired speed from gap error $d - d_r$ (with **time headway** $d_r = d_0 + \tau_h v$), the inner PID tracks speed.

### 2.2 Longitudinal MPC

Cast speed planning as optimization (especially when speed limits, grade, cut-ins impose many constraints):

$$
\min_{a_{0:H-1}}\ \sum_{k=0}^{H-1}\!\Big[q\,(v_k - v_{r,k})^2 + r\,a_k^2 + r_\Delta\,(a_k-a_{k-1})^2\Big]
\quad\text{s.t.}\quad
\begin{cases}
s_{k+1}=s_k+v_k\Delta t\\
v_{k+1}=v_k+a_k\Delta t\\
\underline{a}\le a_k\le\bar a,\ \ |a_k-a_{k-1}|\le \Delta a_{\max}\\
s_k \le s_{\text{lead},k}-d_{\text{safe}}
\end{cases}
$$

The jerk term $(a_k-a_{k-1})^2$ directly optimizes **comfort (jerk)**; the lead-vehicle safe distance, as a hard constraint, naturally handles cut-ins. This is a convex QP, solvable in a few ms.

**Curvature derivation**: let the tracking circle pass through the current rear-axle point and the lookahead point. Its center lies on the body's lateral axis (perpendicular to heading), at distance $R$ from both. The chord to the lookahead point has length $L_d$ and makes angle $\alpha$ with the heading. By circle geometry (chord $L_d$ subtends central angle $2\alpha$):

$$
\frac{L_d}{\sin(2\alpha)} = 2R
\;\Longrightarrow\;
\frac{L_d}{2\sin\alpha\cos\alpha}=2R
$$

Noting the lateral offset $e_y = L_d\sin\alpha$ and longitudinal $L_d\cos\alpha$, we directly obtain the more common form:

$$
\boxed{\ \kappa = \frac{1}{R} = \frac{2\sin\alpha}{L_d}\ }
\qquad\Longrightarrow\qquad
\delta = \arctan(\kappa L) = \arctan\!\Big(\frac{2L\sin\alpha}{L_d}\Big)
$$

**The key tuning knob is the lookahead distance $L_d$**, usually speed-dependent $L_d = k_v v + L_0$: short lookahead at low speed (precise, hugs the line), long lookahead at high speed (smooth, non-oscillatory). Too short → high-frequency oscillation; too long → corner cutting. Pure Pursuit uses geometry only, with no heading-error term, so it has steady-state offset on high-curvature paths.

### 3.2 Stanley Control

The lateral controller of Stanford's DARPA-winning car, referenced at the **front axle**, correcting both the **cross-track error $e$** and the **heading error $\theta_e$** at once:

$$
\delta = \theta_e + \arctan\!\Big(\frac{k\,e}{v + k_s}\Big)
$$

- The first term $\theta_e = \theta_r - \theta$ aligns the heading with the path tangent;
- The second is nonlinear feedback: when cross-track error is large, the angle approaches $\pm\pi/2$ (drives straight at the path); when small, it converges linearly. The denominator $v$ auto-reduces gain at high speed (avoids high-speed shimmy), and $k_s$ prevents division by zero at low speed.

One can show that under the bicycle model $\dot e \approx -\,kv\cdot e/(v) = -k e$, i.e. exponential convergence. Stanley hugs the line better than Pure Pursuit (no steady-state cross-track offset) but is more sensitive to cross-track measurement noise and has no explicit curvature feedforward.

### 3.3 LQR Lateral Control

Linearize the lateral dynamics into $\dot{\mathbf{x}} = A\mathbf{x} + B\delta$ with state $\mathbf{x}=[e_y,\ \dot e_y,\ e_\theta,\ \dot e_\theta]^\top$ (cross-track error, its rate, heading error, its rate). Design a quadratic cost:

$$
J = \int_0^\infty (\mathbf{x}^\top Q\,\mathbf{x} + R\,\delta^2)\,dt
$$

Solve the continuous Riccati equation $A^\top P + PA - PBR^{-1}B^\top P + Q = 0$ for the optimal feedback $\delta = -K\mathbf{x}$, $K = R^{-1}B^\top P$. In practice add a **curvature feedforward** $\delta_{ff} = \arctan(\kappa_r L)$ to remove steady-state error in curves. LQR explicitly trades tracking accuracy ($Q$) against control smoothness ($R$), more principled than hand-tuned Pure Pursuit/Stanley, yet still unconstrained and single-point-linearized. See [Robot Control Theory](../../00_Robotics/02_Classical_Robotics/控制理论.en.md).

### 3.4 Lateral MPC

Put the cross-track error dynamics into a receding-horizon optimization, with **explicit constraints** on steering angle and steering rate:

$$
\min_{\delta_{0:H-1}}\sum_k \big(e_{y,k}^2 q_1 + e_{\theta,k}^2 q_2 + \delta_k^2 r + \Delta\delta_k^2 r_\Delta\big)
\quad\text{s.t.}\quad |\delta_k|\le\bar\delta,\ |\Delta\delta_k|\le\Delta\bar\delta
$$

MPC naturally handles actuator physical limits and curvature lookahead (it can "see" an upcoming curve and pre-steer), and is the mainstream lateral controller in production advanced driver assistance.

## 4. Lateral-Longitudinal Coupling and Unified MPC

The previous sections decouple lateral and longitudinal, but physically they couple: in high-speed cornering, longitudinal braking enlarges the tire's lateral-force margin ("brake before the corner"); tire grip is a shared finite resource across both (**the friction circle** $\sqrt{F_x^2+F_y^2}\le\mu m g$). The unified approach uses one **nonlinear MPC** that does trajectory tracking directly on the bicycle (or dynamic) model:

$$
\min_{\mathbf{u}_{0:H-1}}\ \sum_{k=0}^{H-1}\!\Big[\|\mathbf{p}_k - \mathbf{p}_{r,k}\|_Q^2 + \|\mathbf{u}_k\|_R^2 + \|\Delta\mathbf{u}_k\|_{R_\Delta}^2\Big] + \|\mathbf{p}_H-\mathbf{p}_{r,H}\|_{Q_f}^2
$$

$$
\text{s.t.}\quad
\mathbf{x}_{k+1}=f(\mathbf{x}_k,\mathbf{u}_k),\quad
\mathbf{u}_k=[\delta_k, a_k],\quad
\underline{\mathbf{u}}\le\mathbf{u}_k\le\bar{\mathbf{u}},\ \
\sqrt{a_{x}^2+a_{y}^2}\le\mu g,\ \
\mathbf{x}_0=\mathbf{x}_{\text{meas}}
$$

where $f$ is the discretized bicycle/dynamic model of §1, $\mathbf{p}=(x,y)$, and the friction-circle constraint encodes lateral-longitudinal coupling into the feasible set. This is the canonical application of [MPC](../../00_Robotics/02_Classical_Robotics/MPC.en.md) — **receding horizon, execute only the first step, re-solve every step**. Using Frenet coordinates (along-path $s$, lateral $e_y$) makes "track the trajectory" more natural and separates lateral/longitudinal weights in the curvilinear frame. The reference $\mathbf{p}_{r}$ and $v_r$ come from the planner ([Planning and Decision-Making](../03_Prediction_Planning/规划与决策.en.md)).

## 5. Actuation and Stability

The control law is elegant on paper; deployment is tormented by three things repeatedly:

### 5.1 Latency

From camera/radar to steering taking effect there is 100–300 ms of pipeline latency (perception, planning, CAN bus, steering-actuator mechanical response). Latency reduces closed-loop phase margin and can destabilize, especially at high speed. Remedies:

- **Smith predictor / state extrapolation**: use the model to roll the current state forward by one delay period and control on the "future state".
- **Built-in delay model in MPC**: write the actuator first-order lag $\dot\delta = (\delta_{cmd}-\delta)/\tau$ into $f$.

### 5.2 Saturation

Physical limits: $|\delta|\le \delta_{\max}$ (about ±30–40° at the road wheel), $|\dot\delta|\le\dot\delta_{\max}$ (steering rate), and acceleration/jerk limits. Once an unconstrained controller's (PID/LQR) command exceeds a limit and gets hard-clipped, the closed-loop behavior departs from the design. MPC handles these as hard constraints natively; PID/LQR need explicit clamping + anti-windup.

### 5.3 Anti-Windup

A PID integral term keeps accumulating while the actuator is saturated (command already at the limit but the error persists); when saturation clears it causes a huge overshoot. Standard remedies:

- **Clamping / conditional integration**: freeze the integrator while saturated.
- **Back-calculation**: feed "command − actual saturated output" back into the integrator, $\dot I = K_i e + K_t(u_{sat}-u)$.

Plus: rate limiting to protect the mechanism, smoothing filters to suppress chatter, and a **fallback** when MPC finds no feasible solution (degrade to Pure Pursuit + safe deceleration).

## Method Comparison

| Method | Error used | Constraint handling | Lookahead | Compute | Best fit |
|---|---|---|---|---|---|
| **Pure Pursuit** | lookahead-point geometry (lateral) | none | implicit (via $L_d$) | very low | low speed, low curvature, limited compute; parking/campus |
| **Stanley** | cross-track + heading error | none | none | very low | low-to-mid speed line tracking (DARPA off-road) |
| **LQR** | 4-dim error state | none (needs clamp) | single-point linearization | low | steady speed, known-model structured roads |
| **MPC** | full state + trajectory | **hard constraints (steer/accel/friction circle)** | explicit H steps | high (QP/NLP) | high speed, tight constraints, lat-long coupling; production ADAS |
| **Learning / E2E** | pixels→control | implicit | learned | medium inference | complex urban, hard-to-model scenes (research/frontier) |

Rule of thumb: tight compute + low speed → Pure Pursuit/Stanley; need steady-state accuracy + known model → LQR + curvature feedforward; actuator constraints / high speed / lat-long coupling → MPC.

## Common Pitfalls

- **Running the kinematic model at high speed**: the low-speed no-slip assumption systematically biases the controller under high lateral acceleration, causing understeer. At high speed, switch to the dynamic model or estimate cornering stiffness online.
- **Hard-coding Pure Pursuit lookahead**: a fixed $L_d$ oscillates at low speed and cuts corners at high speed. Must adapt $L_d = k_v v + L_0$ with speed.
- **Ignoring actuation latency**: a PID/LQR perfect in simulation shimmies at high speed on the car — usually because 150–300 ms of pipeline latency is uncompensated.
- **PID without anti-windup**: after full throttle uphill / steering wound to the lock, integral windup causes violent overshoot upon release.
- **Fully decoupling lateral and longitudinal**: in a curve the lateral MPC steers hard while the longitudinal does not decelerate, exceeding the friction circle and sliding. Shared grip needs a joint constraint.
- **No fallback when MPC is infeasible**: overly tight constraints or ill-conditioned initialization make the solver fail; there must be a degraded controller and safe deceleration as a backstop, or the steering "locks up".
- **Stanley division-by-zero at low speed**: the $k_s$ in the denominator $v+k_s$ cannot be dropped, or the angle diverges near a stop.
- **Un-normalized cost weights**: inconsistent units in $Q,R$ (meters vs radians) make tuning a black art; normalize by physical scale first.

## Frontier Trends

- **Differentiable / learning-augmented MPC**: embed MPC as a differentiable layer in a network to learn cost weights and dynamics residuals end-to-end, combining constraint guarantees with data adaptivity.
- **End-to-end control**: from [ALVINN](https://en.wikipedia.org/wiki/ALVINN)(1989), PilotNet(2016) to UniAD(2023), VAD(2023) mapping sensors directly to trajectory/control; see [End-to-End Autonomous Driving](../05_Learning_E2E/端到端驾驶.en.md). The control layer is gradually absorbed into "trajectory planning + a lightweight tracker" or a full-stack neural net.
- **World-model-driven control**: MILE(2022), GAIA-1(2023) learn an onboard world model and roll out in imagination for model-based control / evaluation — a neuralized version of MPC.
- **Contact / limit-handling control**: race-grade MPC (drifting, at the grip limit) with real-time tire-force estimation, pushing the friction-circle constraint to its boundary.
- **Safety filters**: wrap a learned policy with a minimally-invasive safety layer using control barrier functions (CBF) / reachable sets, combining performance with provable safety.
